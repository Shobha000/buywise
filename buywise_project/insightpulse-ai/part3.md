## CHAPTER 7: TESTING STRATEGIES

### 7.1 Unit Testing
The foundation of the BuyWise testing strategy lies in rigorous, isolated unit testing, specifically targeting the data validation and extraction layers to ensure absolute data integrity. On the backend, extensive tests were written against the Pydantic models (such as `ReviewOut`) to verify schema compliance. These tests specifically push edge cases into the models to ensure complete null safety; for example, verifying that the system does not crash if an optional field like `images` or `sentiment_score` returns as `None` due to a scraping anomaly. A secondary, critical phase of unit testing focused exclusively on the Scraper boundary logic. Given the volatility of DOM structures, unit tests were engineered to validate the `BeautifulSoup` traversal paths. These tests feed static HTML snippets into the scraper to ensure that the DOM walker correctly identifies the strict boundaries of a single review container. This guarantees that the algorithm stops exactly at the end of a specific review, completely preventing the catastrophic bleeding of author names, star ratings, or images into adjacent review records within the database.

### 7.2 Integration Testing
While unit tests verify isolated components, integration testing was critical to ensure that the complex, decoupled architecture of BuyWise functions cohesively under simulated real-world conditions. The most intensive integration test focused on the resilience of the WebSocket feed. To simulate a heavy load, an automated script was engineered to force the rapid insertion of 100 simultaneous, fully analyzed review objects directly into the database. The objective was to verify that the FastAPI `ConnectionManager` successfully broadcast all 100 payloads, and more importantly, that the React frontend could parse, deduplicate, and render them instantly without dropping frames, locking the UI, or duplicating DOM nodes. Another vital integration test validated the database-level deduplication logic. The system was subjected to consecutive searches for the exact same product (e.g., searching "iPhone 15" three times in a row). The test successfully proved that the backend accurately recognized the previously scraped texts, actively skipping the database insertion phase, and verifying that the unique `id` counter in SQLite did not increment, thereby preventing massive database bloat during repeated user queries.

### 7.3 Performance Testing
Performance testing for BuyWise shifted the focus away from internal code execution speed and directly toward the system's ability to interact with hostile, heavily defended external environments under sustained pressure. The primary focus of this testing phase was validating the resilience of the Scraper engine. Because massive e-commerce platforms actively deploy sophisticated anti-bot countermeasures, standard HTTP requests fail almost immediately. The performance tests heavily scrutinized the `curl_cffi` implementation, pushing the system to execute rapid, concurrent scrapes across Amazon and Flipkart. The goal was to ensure that the randomized TLS fingerprinting effectively bypassed basic HTTP 403 blocks and CAPTCHA walls over extended periods. The tests confirmed that the architecture successfully maintains a high success rate during heavy, sustained search loads. Furthermore, performance profiling on the backend verified that the asynchronous architecture prevents the heavy I/O wait times of the scraper from blocking the ASGI server, allowing the API to remain highly responsive to other client requests even while multiple scraping tasks are actively resolving in the background.

## CHAPTER 8: LIMITATIONS

Despite its advanced architecture, the BuyWise platform currently operates under several notable technical limitations. The most significant limitation stems from the platform's reliance on static HTML DOM parsing. Because the current scraping modules utilize libraries like `BeautifulSoup`, they are entirely incapable of executing JavaScript. Consequently, reviews that are hidden behind deep React or Vue asynchronous loads, or media that requires user interaction to render, cannot be extracted by the current system without integrating heavy headless browsers (like Playwright). This, however, would drastically increase server computational costs and latency. A secondary limitation involves external IP Rate Limiting. While the system successfully utilizes TLS fingerprinting to bypass basic bot identification scripts, the system still routes all traffic through a single, static IP address. Aggressive, repetitive scraping from this single IP can trigger behavior-based heuristics on Amazon or Flipkart, leading to temporary IP bans. Finally, there is a strict limitation regarding the size and capability of the Machine Learning models. While deploying state-of-the-art HuggingFace transformer models (such as BERT or RoBERTa) would yield near-perfect fake review detection, running these large parameter models locally requires significant GPU memory (VRAM). This restricts the current local deployment to utilizing lighter, heuristic-based mathematical models, which, while exceptionally fast, lack the deep semantic understanding of full transformer architectures.
